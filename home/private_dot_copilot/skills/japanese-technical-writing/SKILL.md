# 日本語技術文書の文章規範

**文章規範スキル**：日本語の技術文書を、論理が追いやすく、読み手の負荷が低く、過剰な演出のない文章へ整える。

使用する場面：日本語の技術書原稿、技術記事、解説文、README、ADR、設計文書、運用手順、レビューコメント、提案書の執筆、推敲、リライト、文章レビュー。
使用しない場面：カジュアルな雑談、英語文書だけの編集、コードそのものの設計判断。

このスキルは https://gist.github.com/k16shikano/fd287c3133457c4fd8f5601d34aa817d の文章規範を、Copilot CLI で使うための運用チェックリストとして整理したものである。

## 基本ルール

- 段落の区切りは空行で示す。段落内の改行は、媒体や既存文書の形式に合わせて必要な場合だけ使う。
- コード、差分、ログ、設定断片はコードブロックに入れる。
- 本筋から外れる由来、命名、補足は脚注へ降ろす。
- 用語を初めて定義するときは太字にする。以後の言及、引用、通称は「」で示す。
- 定義の箇条書きは `**用語**：説明` の形にする。
- 日本語の地の文や見出しでダッシュ類を使わない。補足は括弧、言い換えは文の分割か読点で処理する。ただし、範囲を示す en ダッシュ、英語の複合語（`Curry–Howard` など）、コードブロック、書誌情報は対象外とする。
- 日本語の並列に中黒を使わない。ただし固有名詞の一部なら使ってよい。
- 見出しは内容を特定できる自然な句にする。区切り線で二要素を詰め込まず、結論を決め台詞として言い切らない。
- 見出しは、その節が答える問い、または扱う対象を指すものにする。「例に戻す」「読み直す」のような作業手順だけの見出しにしない。
- 見出しは疑問形でも名詞句でもよい。本文のトーンに合わせ、対象や問いが分かる形を選ぶ。

## 段落と論証

- 一つの段落には一つのトピックだけを置く。
- 段落の最初の文で、その段落の話題を示す。
- 段落の先頭で、前段落との関係を必要に応じて明示する。
- 論証は一方向に進める。疑念や反論を処理してから、結論を一度だけ置く。
- 読者が誤解しそうな命題は「」で正確に書き出し、それを否定してから本当の理由を述べる。
- 「AではなくB」と書くときは、Aを否定する根拠を添える。
- 譲歩では事実の確認にとどめる。あとで訂正する内容を著者の声で因果として断定しない。表面的な診断を認めるときは、読者や通説の声に帰属させる。
- 例が作為的に見える場合は隠さない。読者が持ちそうな疑念を認め、現実にも起きうる根拠を読者の経験や通説に求める。
- 前方参照は段落末や節末へ置き、論証の途中に挟まない。

## 論理の厳密さ

- 推量、可能性、読者の疑念、作中人物の認識、反実仮想として必要な弱い表現は断定へ変えない。
- 異なる決定、原因、問題を「同じ」とまとめない。
- 複数の要因がある事象を単一原因に還元しない。
- 章や節をまたいで、同じ概念の分類、定義、用語を一致させる。
- 因果を主張するときは、なぜそうなるのかを一文で示す。
- 検出、保証、解決を無条件に約束しない。条件や範囲を明示する。
- 例が主張全体を支えているか確認し、支えていないなら主張を狭める。
- 「次節で扱う」と書いた論点は、実際にそこで回収する。
- 「ただし」「とはいえ」で譲歩や限定を置いたら、その後で論を進める。逆接で終えて宙吊りにしない。
- 確認していないことを、確認したかのように書かない。

## 規則の判断を支える例

- 推量の保持。悪い例：「提示し続けているかもしれない」を根拠なく「提示し続けている」に変える。良い例：「提示し続けている可能性がある」のように、不確実性を残して整える。
- 対象の腑分け。悪い例：相互依存する三つの未決事項を「同じ決定を別々に下していた」とまとめる。良い例：「どれも別々の決定であり、しかも互いに依存している」と切り分ける。
- 単一原因への還元を避ける。悪い例：契約の不在と情報隠蔽の失敗が混ざった事故を、丸ごと「情報隠蔽の問題」と説明する。良い例：どの問題をどの概念で説明するのかを対応づける。
- 因果の機構を示す。悪い例：「手順で分けると変更が全体に波及する」。良い例：「各工程がデータを受け渡すための表現を共有してしまい、その表現を変えると全体に波及する」。

## 読者の負荷

- 後で参照しないファイル名、関数名、識別子、時刻、数値を出さない。
- 抽象語が何を指すか曖昧なときは、丸括弧でその場で特定する。
- 新しい例を追加するときは、前の例との違いと必要性を先に示す。
- 導入部には、後の問いや帰結に関係しない詳細を詰め込まない。
- 例示では受動態の結果羅列を避け、行為者を主語にした動作の連なりで書く。
- 架空の人物設定を無意味に冠しない。「入社2年目のエンジニアが」のような設定は、議論に必要な場合だけ使う。
- 読者をむやみに「あなた」と呼ばず、「開発者」「読者」など役割名で書く。ただし、場面への導入（「としよう」）や章や本の結びでは二人称を使ってよい。
- 「AI」「ツール」のような広すぎる語でぼかさず、対象を具体的に書く。
- 術語や訳語は分野で慣用される語を選ぶ。
- 人物そのものに言及するときは原綴りで書く。ただし、歴史上の人物や人名を冠した概念を定着名で紹介するときは、日本語で通用しているカタカナの通称を使う。

## 演出の節度

- 演出の規範は全面禁止ではなく、節度の規範である。修辞は、それが効果を生む箇所でのみ使う。
- 溜め、修辞疑問、短い決め台詞は多用しない。
- 太字強調は、誤読を防ぐ否定や節の帰結など論理の要所に限る。
- 「重要なのは」のような前置きで主張を予告せず、主張をそのまま書く。
- 「AではなくBだった」という対句を多用しない。
- 命令調の断定より、作業者の判断として書く形を選ぶ。
- 指す内容が一意に決まらない比喩を避け、平易な動詞で書く。
- 事故や危険を煽るためだけの列挙をしない。

## LLM 風の空句を避ける

次の表現は、論点を増やさない装飾になっていないか疑う。必要な場合を除き、具体的な主張へ置き換える。

- 「重要なのは」「本章では」「ここでは見ていく」「まとめると」「に他ならない」
- 直前の言い換えでしかない「要するに」。ただし、数文の議論を一文へ圧縮する要約の合図としては使ってよい。
- 「正面から扱う」「正面から回収する」
- 「不可欠」「核心的」「鍵となる」「根本的」「多角的」「包括的」「総合的」
- 「掘り下げる」「深掘りする」「言語化する」「触れる」「言及する」
- 「の観点から」「という側面から」「において」
- 「さらに」「また」「加えて」の連打
- 根拠なく弱める「と言えるだろう」「かもしれない」
- 語調を整えるための意図的な緩和は許す。削るのは、根拠なく主張を弱めている表現に限る。
- 中身のない強調としての「非常に」「極めて」「大いに」

例：

- 悪い例：「本章では、対象の理論を正面から扱う」。良い例：「本章では、対象の理論を扱う」。
- 悪い例：「多角的に分析すると、重要なのは設計の見直しである」。良い例：「評価の焦点は、正しさを誰が知っているかにある」。

## 冗長さの点検

- 同じ主張を言い換えて繰り返さない。
- 隣接する節の役割が重複しているなら統合する。
- 場面描写の直後に内容を要約し直さない。
- 同じ論理的役割の事実は、一文にまとめられないか確認する。
- 読者が補える中間説明は削る。
- 接続や評価のためだけの文を置かない。
- 文章のリズムを作るための接続表現は、冗長と見なさない。
- 想像上の読者との問答を修辞として使わない。
- 著者の立場の弁明ではなく、事実と根拠を書く。

## レビュー手順

1. 対象文書の目的、読者、範囲を確認する。
2. 段落区切り、コードブロック、脚注、用語定義、見出し形式を整える。
3. 段落ごとにトピック文、前段落との関係、論証の向きを確認する。
4. 主張、例、因果、限定、前方参照の整合性を確認する。
5. 読者が保持する固有名、数値、例、抽象語を減らす。
6. 過剰な演出、LLM-like filler、冗長な繰り返しを削る。
7. 推量や不確実性を、根拠なく断定へ変えていないか最後に確認する。
